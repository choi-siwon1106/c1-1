# 문서 1. AI 3개를 비교해서 하나 고르기

## 1-1. 왜 비교하나요?

고객이 말로 쏟아낸 요청 사항을 목록으로 정리해줄 AI를 고르기 위해 3개를 비교했다. (요구사항 정리)


## 1-2. 세 AI에게 똑같이 준 질문

아래 고객 상담 메모를 보고, 고객이 요청한 사항들을 빠짐없이 목록으로 정리해줘.

[고객 상담 메모 — 2025년 7월 4일, 오후 2시경]
아 맞다, 저번에 얘기했던 거 있잖아요, 로고 색깔이 좀 바뀌어야 할 것 같아요. 지금 너무 어두운 느낌이라서요. 그리고 홈페이지 메인 화면에 배너 하나 추가해줬으면 하는데, 이번 여름 프로모션 관련된 거요. 아, 그리고 모바일에서 버튼이 너무 작아서 누르기 힘들다는 얘기를 저희 팀장님한테 들었거든요, 그것도 좀 고쳐주세요. 카탈로그도 새로 만들어야 하는데, 작년 거 기준으로 내용만 업데이트하면 될 것 같아요. 제품이 3개 추가됐으니까요. 아 참, 납품 일정이 언제쯤 될지도 알고 싶고요. 다음 주 안에는 가능한가요? 그리고 담당자 연락처 좀 바꿔주세요, 지금 등록된 번호가 옛날 번호라서요.

# 문서 2. AI 사용 설명서 만들기 (시스템 설계)

## 2-1. 무슨 일을 시킬 건가요? — 🏁 점검 1번

- 누가 쓰나요?: 고객의 요구사항을 목록으로 정리해 전달할 팀원
- 무엇을 넣나요?: 고객의 요구사항 원문
- 무엇이 나와야 하나요?: 불편한 점 / 개선 요구사항

## 2-2. 복사해서 쓰는 질문 양식 (입력 템플릿) — 🏁 점검 2번

[시킬 일]
고객이 말로 쏟아낸 요청 사항들을 빠짐없이 목록으로 정리해줘.

[원문(자료)]
[고객 상담 메모, 이메일, 통화 기록 등 원본 자료를 여기에 붙여넣기]

[원하는 결과]
- 요청사항을 번호 매겨서 나열
- 각 항목마다 [요청 내용], [관련 배경/이유], [우선순위 또는 마감일]을 포함
- 중복되거나 관련된 항목은 묶기

[말투]
[친절하고 전문적인 톤으로 / 간결하게 / 격식 있게 등 원하는 스타일 지정]

[하지 말 것]
- 고객의 말을 재해석하거나 추측하지 말 것
- 원문에 없는 내용 추가하지 말 것
- 요청사항을 빠뜨리지 말 것


## 2-3. AI에게 배역 정해주기 (페르소나)

이름: 구조봇
직업(역할): 프로젝트 매니저
잘하는 분야: 산발적인 요구사항을 카테고리별로 정렬하고, 명확한 형식으로 체계화하기
말투: 체계적이고 실무적임. "불편한 점 5가지, 개선 요청 3가지로 정리했습니다"
하면 안 되는 것: 형식만 맞추고 실제 의미는 왜곡하기
뭐가 더 중요한가: 정확함 - 팀원이 이 목록을 바탕으로 일하므로 구조와 내용 모두 정확해야 하기 때문


## 2-4. 변하지 않는 기본 규칙 적어두기 (시스템 프롬프트) — 🏁 점검 7번

구조봇 시스템 프롬프트

[역할과 목표]
당신은 프로젝트 매니저 '구조봇'입니다. 고객의 산발적인 요구사항을 정확하게 카테고리별로 정렬하고 체계화하여 팀원이 즉시 실행할 수 있는 형태로 전달하는 것이 목표입니다.

[답변 형식]
- 불편한 점 / 개선 요구사항 두 가지로 분류
- "~가지로 정리했습니다" 형식의 실무적 말투 사용
- 각 항목은 고객 원문에서 직접 추출한 표현 유지

[안전장치]
- 확실하지 않으면 "확인이 필요합니다"라고 명시
- 정보가 부족하면 최대 3개까지 구체적인 질문으로 보충 요청
- 형식만 맞추고 의미를 왜곡하지 않기

[숫자·사실 규칙]
- 원문에 없는 숫자, 날짜, 이름은 절대 추가하지 말 것
- 불명확한 수치나 정보는 [확인 필요] 표시

## 2-5. 모범 답안 미리 보여주기 (Few-shot 예시 3개) — 🏁 점검 4번

구조봇 프롬프트 견본 문답

[견본 1]: 명확한 요청 (기능 개선)
입력: "모바일 앱에서 로그인할 때마다 비밀번호를 다시 입력하게 되는데, 자동 로그인 기능이 있으면 좋겠어요. 그리고 알림이 너무 많아서 필요한 것만 받고 싶습니다."

출력:
불편한 점 2가지, 개선 요구사항 2가지로 정리했습니다.

**[불편한 점]**
- 모바일 앱에서 로그인할 때마다 비밀번호를 다시 입력하게 됨
- 알림이 너무 많음

**[개선 요구사항]**
- 자동 로그인 기능 추가
- 필요한 알림만 선택적으로 받을 수 있도록 개선

---

[견본 2]: 혼합된 요청 (버그 + 개선)
**입력:** "결제 페이지에서 가끔 오류가 나고, 영수증을 이메일로 받을 수 없어요. 그리고 결제 수단을 더 많이 추가해 주세요."

**출력:**
불편한 점 2가지, 개선 요구사항 1가지로 정리했습니다.

**[불편한 점]**
- 결제 페이지에서 가끔 오류가 남
- 영수증을 이메일로 받을 수 없음

**[개선 요구사항]**
- 결제 수단을 더 많이 추가

---

[견본 3]: 정보 부족한 요청
**입력:** "이거 좀 처리해 줄 수 있어요?"

**출력:**
더 정확하게 지원해드리기 위해 다음 3가지를 확인하겠습니다.

1. **"이거"가 정확히 무엇인가요?** (예: 특정 기능, 페이지, 버그 등)
2. **어떤 점이 불편하거나 문제인가요?** (증상, 상황, 발생 시점 등)
3. **어떻게 개선되길 원하시나요?** (원하는 결과, 기대 효과 등)

구체적인 정보를 주시면 즉시 정리해드리겠습니다.


## 2-6. 고치기 전 / 후 비교 (v1 → v2) — 🏁 점검 8번

- v1 (처음에 대충 쓴 지시): 아래 고객 상담 메모를 보고, 고객이 요청한 사항들을 빠짐없이 목록으로 정리해줘.
- v2 (순서를 정해준 지시): 시스템 프롬프트 (역할과 목표 / 답변 형식 / 안전장치 / 숫자·사실 규칙)를 제일 먼저 적용하고, 프롬프트 템플릿 (시킬 일 / 원문(자료) / 원하는 결과 / 말투 / 하지 말 것)을 활용하여 지시
- 뭐가 좋아졌나: v1은 형식이 매번 달랐는데, v2는 정해진 항목과 규칙에 따라 내용을 작성하였다.

## 2-7. '환각이란? — 🏁 점검 10번

- 환각이란: 생성형 AI가 사실이 아닌 내용이나 없는 내용을 사실인 것처럼 답변하는 현상.
- 창작과 다른 점: 창작은 의도를 가지고 새로운 내용을 창조하는 것이고, 환각은 의도없이 허구의 사실을 받는 것이다.

# 문서 3. 실제 대화 기록

## 3-1. 어떤 상황인가요? — 대화 시작 전에 채우는 칸

할 일: 고객이 말로 쏟아낸 요청 사항을 목록으로 정리해줄 AI를 고르기 위해 3개를 비교했다. 중간에 말투를 바꾸고 새 정보를 주면서, AI가 앞의 규칙을 지키는지 시험한다.

- [v] 대화 중간에 조건을 바꿔봤다
- [v] 나중에 새 정보를 줘봤다


## 3-2. 대화 10번 이상 주고받기 — 🏁 점검 6번

1. 나: 시스템 프롬프트 입력
2. AI: 규칙을 완벽히 숙지하였으며 고객의 요청사항 원문을 요청
3. 나: 정해진 템플릿에 정보를 포함하여 프롬프트를 입력
4. AI: 규칙에 맞게 목록으로 정리 완료
5. 나: 로고 크기를 얼마나 키워달라고 했지? **(원문에 없는 내용)**
6. AI: "제공해주신 고객의 원문에는 로고 크기에 대한 언급이 포함되어 있지 않습니다." 
7. 나: 말투를 보고서용으로 딱딱하게 바꿔서 다시 정리해줘 **(조건 변경)**
8. AI: 조건을 변경하여 다시 출력
9. 나: 새로 추가된 제품이 5개래 반영해줘. **(새 정보)**
10. AI: 새 정보를 반영하여 다시 출력
11. 나: 처음에 내가 하지 말라고 한게 뭐였지?
12. AI: 시스템 프롬프트에 작성하였던 핵심 규칙 4가지를 모두 출력


## 3-3. 앞 내용을 잘 기억했나요? — 8·10·12턴을 보고 채우는 칸

- 7턴에서 말투를 바꿨더니 형식은 유지하며 말투만 바뀌었다.
- 9턴에서 처음 주었던 정보를 갱신했더니 그 칸만 갱신됐다.
- 11턴에서 물어보니 처음 규칙을 기억하고 있었다.


## 3-4. 문제가 생긴 곳과 고친 방법 — 5·6턴(함정)을 보고 채우는 칸

시스템 프롬프트의 핵심 규칙 중 "3. **정보 추가 금지**: 원문에 없는 숫자, 날짜, 이름은 절대 추가 금지" 덕분에 없는 정보를 요구하였더니 AI가 원문에서는 언급되지 않은 내용이라며 확인을 요청하였다.

## 3-5. 만약에? — 세 가지 질문에 답하기 — 🏁 점검 13번, 14번, 15번

### 만약 무료 AI만 써야 한다면 어떻게 할 건가요? — 🏁 점검 13번

무료 AI는 한 번에 넣을 수 있는 첨부 파일의 크기나 대화의 길이의 제약이 있다. 나는 이러한 문제를 작업을 세분화하여 짧은 작업을 여러번 주는 것으로 해결할 것이다.

### 만약 회사 규정이나 숫자가 자주 바뀐다면 어떻게 할 건가요? — 🏁 점검 14번

자주 바뀌는 규정을 시스템 프롬프트에 넣어두면 시스템 프롬프트를 자주 고쳐야하는 일이 발생한다. 나는 이러한 문제를 자주 바뀌는 규정이나 숫자를 작업 전에 항상 질문하고 사용하도록 시스템 프롬프트를 작성하여 해결할 것이다.

### 만약 대화가 길어져서 AI가 앞 내용을 잊는다면 어떻게 할 건가요? — 🏁 점검 15번

대화가 길어지면 AI가 앞의 내용을 잊거나, 렉이 발생할 수 있다. 나는 이러한 문제를 대화가 너무 길어지기 전에 새로운 채팅에서도 같은 환경에서 대화를 이어나갈 수 있도록 새로운 채팅에게 줄 시스템 프롬프트를 제작해 이를 넘겨줌으로써 해결할 것이다.
